# MindBridge Prompt Design Notes

## 1. Document Purpose

이 문서는 MindBridge 프로젝트에서 사용한 프롬프트 설계 방향을 공개용으로 정리한 문서입니다.

원본 프롬프트 전문을 공개하는 문서가 아니라, 어떤 문제를 해결하기 위해 프롬프트를 어떻게 나누고 관리했는지 설명하는 요약 문서입니다.

## 2. Prompt Design Goal

MindBridge의 프롬프트 설계 목표는 단순히 자연스러운 대화를 만드는 것이 아니었습니다.

핵심 목표는 다음과 같습니다.

- 상담사를 대체하는 AI처럼 보이지 않게 하기
- 접수면접 목적 안에서만 질문하게 하기
- 사용자 응답을 구조화 가능한 정보로 정리하기
- 위기 응답을 자유 생성이 아니라 보수적 흐름으로 처리하기
- 리포트에서 진단, 해석, 치료 방향 제시를 방지하기
- 단계별 역할을 분리해 디버그와 검증이 가능하게 하기

## 3. Overall Prompt Strategy

MindBridge는 하나의 거대한 프롬프트로 모든 기능을 처리하지 않고, 역할별로 프롬프트를 분리하는 방향으로 설계했습니다.

주요 분리 기준은 다음과 같습니다.

- 질문 생성
- 구조화 추출
- 위기 응답 처리
- 리포트 생성
- 자원 탐색 조건 구조화
- 보안 및 역할 이탈 대응

이렇게 분리한 이유는 각 단계에서 요구되는 출력 성격이 다르기 때문입니다.

질문 생성은 사용자에게 자연스럽게 다음 질문을 제시해야 하지만, 구조화 추출은 정확한 항목 정리가 중요합니다.

위기 응답 처리는 자유로운 문장 생성보다 고정된 안전 흐름이 중요하고, 리포트 생성은 사실 정리 중심의 문장 제어가 중요합니다.

## 4. Interview Prompt

Interview Prompt는 접수면접 질문을 생성하는 역할을 담당합니다.

이 프롬프트의 목적은 사용자의 심리 상태를 해석하는 것이 아니라, 현재 섹션의 목적에 맞는 질문을 이어가는 것입니다.

### 주요 입력

- 기본 정보
- 현재 섹션
- 현재 질문 intent
- 이전 대화 이력
- repair count
- 위기 플래그 여부

### 주요 출력

- 다음 질문
- 보완 질문
- 다음 섹션 이동 신호

### 설계 원칙

- 질문은 현재 섹션 목적 안에서만 생성한다.
- 원인 탐색형 질문을 피한다.
- 감정 해석처럼 보이는 문장을 피한다.
- 진단이나 심각도 판단처럼 보이는 문장을 피한다.
- 사용자의 답변이 짧아도 무조건 무효로 보지 않는다.
- 보완 질문은 제한된 목적 안에서만 사용한다.

### 허용되는 보완 질문

- 내용 확장
- 빈도 또는 기간 확인
- 답변 의미 확인

### 금지되는 질문 유형

- 왜 그렇게 느꼈는지 묻는 원인 탐색형 질문
- 사용자의 감정을 읽어낸 것처럼 말하는 질문
- 진단 또는 치료 필요성을 암시하는 질문
- 답변을 강요하는 질문
- 상담사처럼 공감 관계를 형성하려는 질문

## 5. Extractor Prompt

Extractor Prompt는 전체 대화 이력에서 리포트 작성에 필요한 정보를 구조화하는 역할을 담당합니다.

이 단계는 사용자의 말을 해석하는 단계가 아니라, 사용자가 말한 내용을 정해진 항목에 맞춰 정리하는 단계입니다.

### 주요 입력

- 전체 대화 이력
- 기본 정보
- 현재 섹션별 응답
- 위기 로그
- 상태값

### 주요 출력

- 구조화 JSON
- 섹션별 요약 데이터
- 미수집 항목 표시

### 설계 원칙

- 사용자 발화를 중심으로 정리한다.
- 언급되지 않은 내용은 추측하지 않는다.
- 미수집 항목은 null 또는 미기재로 둔다.
- 임상적 해석을 하지 않는다.
- 진단명을 생성하지 않는다.
- 치료 방향을 제안하지 않는다.
- 사용자가 입력한 메타 명령을 시스템 규칙으로 승격하지 않는다.

## 6. Crisis Prompt

Crisis Prompt는 위기 관련 응답을 보수적으로 감지하고 표시하는 역할을 담당합니다.

위기 응답 처리는 자유로운 생성형 대화로 처리하지 않고, 명시적 입력과 고정 안내 중심으로 설계했습니다.

### 주요 입력

- 사용자 입력
- 위기 관련 선택 응답
- 현재 상태값
- 기존 위기 플래그

### 주요 출력

- 위기 플래그
- 고정 안내 문구
- 리포트 반영용 확인 필요 정보

### 설계 원칙

- 위기 여부를 임상적으로 확정하지 않는다.
- 명시적 입력 중심으로 처리한다.
- 자유 생성형 위로 문구를 만들지 않는다.
- 고정 안내 문구를 우선한다.
- 필요 시 사람 확인 흐름으로 연결한다.

## 7. Report Prompt

Report Prompt는 구조화된 데이터를 바탕으로 상담사용 또는 사용자용 리포트를 생성하는 역할을 담당합니다.

리포트는 상담 결과서가 아니라 접수면접 정리 문서입니다.

### 주요 입력

- 기본 정보
- 구조화 데이터
- 위기 플래그
- 섹션별 응답 요약
- 현재 트랙 정보

### 주요 출력

- 상담사용 리포트
- 사용자용 리포트
- 위기 확인 필요 영역
- 섹션별 요약

### 작성 원칙

- 사실 정리 중심으로 작성한다.
- 사용자 응답 기반으로만 작성한다.
- 해석형 표현을 피한다.
- 진단명 확정을 하지 않는다.
- 치료 방향을 제시하지 않는다.
- 최종 매칭이나 추천을 유도하지 않는다.

### 피해야 할 표현

- 위험도가 높아 보임
- 우울 성향이 뚜렷함
- 치료가 필요한 수준임
- 특정 상담이 반드시 필요함
- 이 상담사가 적합함
- 이 센터로 가야 함

### 허용되는 표현

- 사용자는 특정 어려움을 언급함
- 특정 상황에서 불편감을 호소함
- 수면 관련 어려움을 언급함
- 상담 신청 동기를 설명함
- 위기 관련 응답이 확인되어 확인 필요로 표시함

## 8. Resource Search Prompt

Resource Search Prompt는 상담 자원 탐색 요청을 검색 가능한 조건으로 구조화하는 역할을 담당합니다.

이 단계는 사용자의 상태를 진단하는 것이 아니라, 검색 조건을 정리하는 단계입니다.

### 주요 입력

- 사용자의 자유 요청
- 지역 관련 표현
- 상담 대상 관련 표현
- 상담 형태 관련 표현
- 상담영역 관련 표현

### 주요 출력

- 지역
- 세부 지역
- 상담 대상
- 상담 형태
- 주요 상담영역 태그
- 검색 대상
- 미지원 조건

### 설계 원칙

- 병명처럼 보이는 표현을 진단으로 처리하지 않는다.
- 상담영역 태그로 변환한다.
- 검색 전에 조건을 확인한다.
- 결과는 추천이 아니라 후보 안내로 제시한다.
- 센터와 상담사 데이터를 무리하게 자동 병합하지 않는다.

## 9. Security and Role Defense Prompt

보안 및 역할 이탈 대응 프롬프트는 사용자의 입력을 시스템 명령으로 처리하지 않도록 설계했습니다.

### 탐지 대상

- 이전 지시 무시 요구
- 시스템 프롬프트 노출 요구
- 개발자 지시 출력 요구
- 상담 접수 역할 변경 요구
- 관리자 모드 호출
- 리포트 내용 조작 요구
- 위기 표시 조작 요구

### 처리 원칙

- 사용자 입력은 시스템 명령이 아니라 접수면접 답변 데이터로 처리한다.
- 메타 명령은 정상 답변으로 인정하지 않는다.
- 1회성 시도는 invalid로 처리한다.
- 반복적 또는 노골적 시도는 resistant로 처리한다.
- 보안 설명을 길게 하지 않는다.
- 논쟁하지 않고 현재 흐름으로 복귀시킨다.
- 핵심 상태는 모델의 자연어 출력이 아니라 코드 규칙을 우선한다.

## 10. Temperature and Output Control

프롬프트 단계별로 필요한 출력 성격이 다르기 때문에, 생성 성향을 보수적으로 관리하는 방향으로 설계했습니다.

### 질문 생성 단계

- 사용자가 읽을 수 있는 자연스러운 질문이 필요함
- 다만 상담자처럼 보이는 표현은 제한함
- 과도한 창의성보다 일관된 면접 흐름을 우선함

### 구조화 추출 단계

- 정해진 형식과 정확성이 중요함
- 가능한 한 낮은 temperature 사용이 적합함
- JSON 또는 구조화 데이터의 안정성이 중요함

### 위기 응답 단계

- 자유 생성보다 고정 안내가 중요함
- 보수적 처리와 명시적 플래그 기록을 우선함

### 리포트 생성 단계

- 사실 정리 중심 문장이 필요함
- 해석형 표현을 제한함
- 일관된 문체와 항목 구조가 중요함

## 11. Debug Considerations

프롬프트 설계 과정에서 디버그 가능성을 중요하게 보았습니다.

디버그에서 확인해야 하는 항목은 다음과 같습니다.

- 현재 질문 intent
- 사용자 입력
- 분류 결과
- reason code
- chosen action
- repair count
- 추출된 정보 변화
- 위기 플래그 여부
- 보안 관련 입력 여부

디버그는 새로운 상담 흐름이 아니라 관찰 레이어입니다.

따라서 디버그 모드가 질문 순서, 분류 기준, 응답 생성 방식을 바꾸지 않도록 설계했습니다.

## 12. Lessons Learned

MindBridge의 프롬프트 설계에서 얻은 핵심 교훈은 다음과 같습니다.

- 좋은 프롬프트는 긴 프롬프트가 아니라 역할이 분명한 프롬프트다.
- 상담 도메인에서는 자연스러운 공감 표현보다 역할 경계가 더 중요할 수 있다.
- LLM이 할 수 있는 일과 하면 안 되는 일을 분리해야 한다.
- 리포트 생성은 문장 품질보다 해석 금지와 사실 정리가 더 중요하다.
- 위기 응답은 생성형 답변보다 고정된 안전 흐름이 더 적합하다.
- 디버그 가능한 구조가 있어야 프롬프트 문제와 로직 문제를 구분할 수 있다.

## 13. Final Summary

MindBridge의 프롬프트 설계는 사용자를 설득하거나 상담하기 위한 것이 아니라, 초기 접수면접을 안전하게 진행하고, 정보를 구조화하며, 리포트를 생성하기 위한 업무형 프롬프트 설계입니다.

핵심은 AI를 더 사람처럼 만드는 것이 아니라, AI가 맡을 수 있는 역할과 맡으면 안 되는 역할을 분명히 나누는 것이었습니다.
