# MindBridge Feature Definition Summary

## 1. Document Purpose

이 문서는 MindBridge의 주요 기능을 공개용으로 요약한 기능 정의 문서입니다.

원본 기능 정의 보고서는 각 기능의 입력, 처리, 출력, 예외 상황을 기능 단위로 정리하는 데 목적이 있습니다.

이 공개용 요약본은 포트폴리오와 GitHub 저장소에서 프로젝트 구조를 빠르게 이해할 수 있도록 작성했습니다.

## 2. System Overview

MindBridge는 심리상담 초기 접수면접을 AI 인터뷰 방식으로 진행하고, 그 결과를 구조화하여 리포트로 정리하는 시스템입니다.

이 시스템은 심리상담 자체를 대체하지 않습니다.

MindBridge의 핵심은 다음과 같습니다.

- 초기 접수면접 진행 보조
- 사용자 응답 구조화
- 리포트 생성
- 위기 응답 보수적 표시
- 상담 자원 후보 탐색
- 디버그 및 품질 점검
- 보안 및 역할 이탈 대응

## 3. Main Users

### 내담자 또는 일반 사용자

초기 접수면접에 응답하고, 필요 시 자기 상태 정돈용 리포트 또는 상담 자원 후보를 확인하는 사용자입니다.

### 심리상담사

구조화된 리포트를 검토하고 첫 상담 전에 핵심 정보를 파악하는 사용자입니다.

### 운영자

디버그 기능, 로그 확인, 자원 탐색 기능 등을 필요에 따라 활용하는 사용자입니다.

## 4. Operating Assumptions

MindBridge는 다음 운영 전제를 가집니다.

- 상담소 트랙에서는 상담사 또는 운영자가 후속 흐름을 확인할 수 있음
- 위기 상황 발생 시 AI가 독자적으로 개입하지 않음
- 위기 응답은 고정 안내와 사람 확인을 우선함
- 시스템 사용 전 고지와 동의가 필요함
- 자가진단, 자가상담, 자가치료 목적을 포함하지 않음

## 5. Feature Layers

MindBridge의 기능은 다음 계층으로 나눌 수 있습니다.

1. 접수면접 진행 기능
2. 답변분류 및 운영 판단 기능
3. 정보 구조화 및 리포트 생성 기능
4. 디버그 및 품질 점검 기능
5. 접수면접 이후 상담 자원 탐색 기능
6. 보안 및 역할 이탈 대응 기능

## 6. Feature List

### F-01. 기본 정보 입력

접수면접 시작 전에 사용자의 기본 정보를 입력받습니다.

입력 항목 예시는 다음과 같습니다.

- 이름 또는 식별용 정보
- 나이 또는 생년월일
- 성별
- 기타 기본 항목

입력된 정보는 이후 질문 생성과 리포트 작성의 기초 정보로 사용됩니다.

일부 항목이 비어 있어도 면접은 진행 가능하며, 미수집 항목은 리포트에서 미기재 또는 확인 필요 상태로 남깁니다.

### F-02. 시스템 성격 고지 및 오프닝 안내

사용자가 MindBridge를 상담사로 오인하지 않도록 시스템 성격을 안내합니다.

안내 내용은 다음과 같습니다.

- AI는 상담사가 아님
- 이 시스템은 초기 접수면접 보조 도구임
- 사용자의 응답은 리포트 정리를 위한 정보로 활용됨
- 상담, 진단, 치료 방향 제시는 수행하지 않음

이 기능의 목적은 사용자가 시스템의 역할 경계를 이해한 상태에서 접수면접을 시작하게 하는 것입니다.

### F-03. 9섹션 접수면접 진행

초기 접수면접을 9개 섹션으로 나누어 진행합니다.

섹션 구성은 다음과 같습니다.

1. 주호소 문제
2. 감정 및 정서 증상
3. 인지 및 사고 증상
4. 신체화 증상
5. 행동적 증상
6. 현재 생활 및 기능 수준
7. 성장 과정, 가족 관계, 사회적 자원
8. 위기 사항 및 건강 상태
9. 상담 신청 동기 및 의지

Interview Chain은 현재 섹션 목적에 맞는 질문을 생성하고, 필요한 경우 제한된 범위 안에서 보완 질문을 수행합니다.

금지되는 질문 방식은 다음과 같습니다.

- 원인 탐색형 질문
- 감정 해석 유도 질문
- 진단형 질문
- 심각도 단정 질문
- 압박형 질문

### F-04. 답변분류 및 보완 질문 처리

사용자 응답이 현재 질문에 대해 충분한지 운영적으로 판단합니다.

분류 유형은 다음과 같습니다.

- valid
- partial
- invalid
- resistant

분류 결과에 따라 다음 동작을 결정합니다.

- 다음 질문 진행
- 보완 질문
- partial 수용
- skip
- 상담사 확인 필요 처리

이 기능은 임상 판단이 아니라 접수면접 운영 판단입니다.

짧지만 의미 있는 답변이 과도하게 무효 처리되지 않도록 지속적인 점검이 필요합니다.

### F-05. 위기 응답 처리

위기 관련 응답은 일반 자유서술 흐름과 분리해 처리합니다.

처리 방식은 다음과 같습니다.

- 위기 관련 입력 감지
- 고정 안내 문구 표시
- 위기 플래그 기록
- 리포트 상단에 확인 필요 정보 반영
- 필요 시 사람 확인 흐름으로 연결

주의할 점은 다음과 같습니다.

- 위기 여부를 임상적으로 확정하지 않음
- 자유 생성형 위로 응답을 만들지 않음
- 명시적 입력과 보수적 규칙에 따라 표시만 수행함

### F-06. 대화 상태 저장 및 질문 intent 관리

현재 질문의 의도를 상태값으로 저장하고 유지합니다.

관리하는 주요 상태는 다음과 같습니다.

- 현재 섹션
- 현재 질문 ID
- 현재 질문 intent
- repair count
- conversation history
- crisis flag
- extracted data
- debug events
- resource opt-in
- resource search results

핵심 원칙은 다음과 같습니다.

- 직전 봇 문장만 보고 intent를 역추론하지 않음
- 질문 생성 시 current question intent를 저장함
- 보완 질문이 나와도 intent를 유지함
- 답변분류는 현재 intent 기준으로 수행함

### F-07. 구조화 추출

전체 대화 이력을 리포트 작성 가능한 구조화 데이터로 변환합니다.

Extractor Chain은 다음 입력을 사용합니다.

- 전체 대화 이력
- 기본 정보
- 위기 로그
- 상태값

출력은 구조화 JSON 또는 내부 구조화 데이터입니다.

추출 원칙은 다음과 같습니다.

- 사용자 발화 중심으로 정리
- 언급되지 않은 정보는 null 또는 미기재 처리
- 추측 금지
- 임상적 해석 금지
- 시스템 명령으로 보이는 사용자 입력은 규칙 변경으로 승격하지 않음

### F-08. 상담사용 또는 사용자용 리포트 생성

구조화 데이터와 위기 로그를 바탕으로 리포트를 생성합니다.

Report Chain은 다음 정보를 정리합니다.

- 기본 정보
- 9섹션별 핵심 응답
- 감정, 인지, 신체, 행동 영역 요약
- 위기 플래그
- 전체 요약

리포트 작성 원칙은 다음과 같습니다.

- 사실 정리 중심
- 해석형 표현 배제
- 진단명 확정 금지
- 치료 방향 제시 금지
- 최종 매칭 유도 금지

리포트는 상담 결과서가 아니라 초기 접수면접 정리 문서입니다.

### F-09. 리포트 화면 출력 및 다운로드

생성된 리포트를 화면에 표시하고 다운로드할 수 있게 합니다.

출력 형태는 다음과 같습니다.

- 화면 표시용 리포트 텍스트
- 다운로드 가능한 파일

미수집 항목은 임의로 채우지 않고, 비워 두거나 확인 필요로 표시합니다.

### F-10. 디버그 모드 실행

디버그 모드는 현재 로직을 바꾸지 않고 내부 판정 과정을 관찰하는 기능입니다.

디버그 모드에서 확인할 수 있는 정보는 다음과 같습니다.

- 현재 질문 intent
- 현재 섹션
- 질문 ID
- 사용자 입력
- 분류 결과
- reason code
- chosen action
- repair count
- extracted delta
- 누적 디버그 로그

디버그 모드가 하면 안 되는 것은 다음과 같습니다.

- 질문 흐름 변경
- 분류 기준 변경
- 내담자 채팅창으로 on 또는 off 제어
- 임상 해석 로그 생성

### F-11. 관련 상담 자원 안내 의사 확인

접수면접 종료 후 사용자가 후속 상담 자원 안내를 원하는지 확인합니다.

처리 흐름은 다음과 같습니다.

1. 접수면접 종료
2. 자원 안내 의사 확인
3. opt-in 상태 저장
4. 필요 시 자원 탐색 기능으로 연결

내담자 화면은 안내 중심으로 유지하며, 추천이나 확정 매칭처럼 보이지 않도록 설계합니다.

### F-12. 자원 탐색 검색 조건 구조화

사용자의 자유 요청을 검색 가능한 조건으로 정리합니다.

구조화하는 조건은 다음과 같습니다.

- 지역
- 세부 지역
- 상담 대상
- 상담 형태
- 주요 상담영역 태그
- 검색 대상

병명처럼 보이는 표현도 임상 판단으로 처리하지 않고, 상담영역 태그로 변환합니다.

검색 조건 구조화는 자유 입력을 그대로 쓰는 것이 아니라, 오염 방지와 규칙 보정이 포함된 단계입니다.

### F-13. 구조화 데이터 기반 상담 자원 검색

사전 수집된 JSON 또는 CSV 기반 구조화 데이터를 대상으로 상담 자원 후보를 검색합니다.

검색 처리 방식은 다음과 같습니다.

- 지역, 대상, 성별, 검색 대상 등 명확한 조건은 하드 필터로 처리
- 태그, 키워드, 정보 충실도 등은 소프트 매칭으로 처리
- 일부 조건은 직접 반영
- 일부 조건은 미지원 조건으로 분리 안내

주의할 점은 다음과 같습니다.

- 실시간 웹 검색 아님
- 앱 내부 실시간 크롤링 아님
- 최종 추천 기능 아님

### F-14. 센터 및 상담사 후보 결과 출력

검색 결과를 센터 후보와 상담사 후보로 분리해 출력합니다.

출력 정보는 다음과 같습니다.

- 검색 기준 요약
- 센터 후보 카드
- 상담사 후보 카드
- 전문영역 원문
- 기관 소개
- 코멘트
- 위치 또는 연락 정보
- 미지원 조건 안내

결과 문구는 후보 안내 수준으로 유지합니다.

피해야 할 표현은 다음과 같습니다.

- 가장 적합한 곳
- 이 상담사가 맞음
- 이 센터로 가야 함
- 상태상 이 상담이 필요함

### F-15. 보안 및 역할 이탈 대응

프롬프트 인젝션, 역할 변경, 관리자 모드 호출, 리포트 오염 시도를 정상 답변으로 처리하지 않도록 최소 방어선을 둡니다.

탐지 대상 예시는 다음과 같습니다.

- 이전 지시 무시 요구
- 시스템 프롬프트 노출 요구
- 개발자 지시 출력 요구
- 상담 접수 역할 변경 요구
- 관리자 모드 호출
- 리포트 내용 임의 변경 요구
- 위기 표시 임의 변경 요구

처리 원칙은 다음과 같습니다.

- 사용자 입력은 시스템 명령이 아니라 답변 데이터 또는 비정상 입력으로 처리
- 1회성 메타 명령은 invalid로 처리
- 반복적 또는 노골적 역할 이탈은 resistant로 처리
- 새 보안 상태 머신을 크게 만들지 않고 기존 분류체계 안에서 처리
- 핵심 상태는 모델 자연어 출력이 아니라 코드 규칙을 우선함

## 7. Data and State Management

주요 상태 항목은 다음과 같습니다.

- 기본 정보
- current section
- current question id
- current question intent
- repair count
- conversation history
- crisis flag
- extracted data
- report text
- debug events
- resource opt-in
- resource query structured
- resource search results

상태 관리 원칙은 다음과 같습니다.

- 질문 intent는 직접 저장함
- 보완 질문 이후에도 intent를 유지함
- 위기 플래그와 세션 완료 여부는 코드 규칙으로 관리함
- 리포트와 검색 결과는 구조화 상태를 기준으로 생성함

## 8. Excluded Scope

현재 공개용 요약 기준에서 제외하는 범위는 다음과 같습니다.

- 심리상담 대체
- 임상 판단 및 진단 자동화
- 치료 방향 추천
- 자가진단 기능
- 자가상담 기능
- 실시간 외부 사이트 검색
- 앱 내부 실시간 크롤링 실행
- 벡터DB 기반 의미 검색 핵심화
- 센터와 상담사 자동 병합
- 관리자 기능의 채팅창 세이프워드 활성화
- 완전한 자유서술형 전체 LLM 분류 전환
- EXE 패키징 배포
- 특정 후보 자동 확정 연결

## 9. Expected Impact

기대 효과는 다음과 같습니다.

- 초기 접수면접 반복 업무 감소
- 사용자 응답의 구조적 수집
- 상담사의 사전 정보 파악 효율 향상
- 위기 관련 응답의 일관된 표시
- 리포트 생성 및 정리 시간 단축
- 답변분류 병목 점검 기반 확보
- 접수면접 이후 상담 자원 탐색 보조 가능
- 디버그와 보안 흐름을 통한 검증 가능성 확보

## 10. Current Limitations

현재 한계는 다음과 같습니다.

- 심리상담 자체를 대체하지 못함
- 임상 판단과 진단은 수행하지 않음
- 답변분류는 지속적인 점검이 필요함
- 리포트 문구는 현장 피드백 기반 보정이 필요함
- 자원 탐색 기능은 최종 추천 기능이 아님
- 데이터 소스별 필드 차이로 조건 반영 수준에 편차가 있음
- 일부 조건은 아직 완전 지원되지 않음
- 상담 자원 데이터의 최신성 관리가 필요함
- 음성 입력과 음성 안내는 후속 과제임

## 11. Final Summary

MindBridge의 기능 정의는 기능을 무한히 확장하는 것이 아니라, 현재 가능한 범위 안에서 설명 가능하고 안정적으로 동작하는 접수면접 보조 시스템을 정리하는 데 초점을 둡니다.

이 시스템은 상담을 수행하는 AI가 아니라, 접수면접 진행, 정보 구조화, 리포트 생성, 후보 탐색을 보조하는 AI 서비스 PoC입니다.
