# MindBridge Retrospective

## 1. Project Summary

MindBridge는 AI 기반 심리상담 초기 접수면접 보조 및 리포트 자동화 시스템 PoC입니다.

이 프로젝트는 3주 미니 프로젝트로 진행되었으며, 초기 접수면접을 구조화하고, 사용자 응답을 리포트로 정리하며, 필요 시 상담 자원 후보 탐색까지 연결하는 흐름을 목표로 했습니다.

핵심 방향은 심리상담을 AI가 대체하는 것이 아니라, 접수면접과 정보 정리 업무를 보조하는 것이었습니다.

## 2. Project Context

프로젝트는 LangChain 기반 AI 서비스 구축 과제로 진행되었습니다.

과제의 주요 요구사항은 다음과 같았습니다.

- LangChain Chain 또는 LCEL 구조 활용
- AI 서비스 기획
- 타겟 사용자와 문제 정의
- 핵심 기능 설명
- 한계점과 개선 방향 제시
- 팀원 역할 분담 및 기여도 정리
- 시연 가능한 결과물 제출

MindBridge는 이 요구사항에 맞춰 단순 챗봇이 아니라, 접수면접 엔진, 구조화 추출, 리포트 생성, 자원 탐색, 디버그, 보안 대응까지 포함한 PoC로 정리했습니다.

## 3. My Role

프로젝트에서 담당한 역할은 다음과 같습니다.

- 서비스 기획
- 문제 정의
- 기능 범위 조율
- 프롬프트 설계
- 역할 경계 정리
- 구현 가능 범위와 후속 과제 분리
- 기획서 작성
- 기능 정의 보고서 작성
- 발표자료 구성
- 발표 흐름 정리
- 리스크와 한계 정리
- 내부 의견 조율

특히 집중한 부분은 AI가 어디까지 개입할 수 있고, 어디서 멈춰야 하는지를 구조화하는 것이었습니다.

## 4. What Went Well

### 서비스 정체성을 좁힌 점

초기에는 심리상담 챗봇처럼 보일 수 있는 위험이 있었습니다.

이를 초기 접수면접 보조 및 리포트 자동화 시스템으로 좁히면서 프로젝트의 설명 가능성이 높아졌습니다.

### 역할 경계를 명확히 한 점

AI가 상담사처럼 행동하지 않도록 다음 기준을 정리했습니다.

- 상담 대체 금지
- 임상 판단 금지
- 진단 금지
- 치료 방향 제시 금지
- 최종 추천 금지
- 사실 정리 중심 리포트

이 기준 덕분에 도메인 리스크를 방어할 수 있었습니다.

### LangChain 구조를 기능별로 분리한 점

Interview, Extractor, Crisis, Report 흐름을 분리하면서 각 단계의 역할을 설명하기 쉬워졌습니다.

하나의 프롬프트가 아니라 기능별 체인 구조로 설명할 수 있었던 점이 발표와 문서화에 도움이 되었습니다.

### 디버그와 보안 흐름을 별도로 정리한 점

답변분류, intent 유지, reason code, chosen action 같은 디버그 요소를 정리하면서 시스템의 검증 가능성을 높였습니다.

또한 프롬프트 인젝션, 역할 변경 요구, 관리자 모드 호출 같은 입력을 정상 답변으로 처리하지 않는 최소 방어선도 정리했습니다.

### 발표 피드백이 긍정적이었던 점

발표 이후 3주 미니 프로젝트임에도 메인 프로젝트급으로 만들었다는 피드백을 받았습니다.

추후 본 프로젝트로 연장해도 될 것 같다는 의견도 받았습니다.

이는 서비스 기획, 기술 구조, 리스크 관리, 발표자료가 하나의 흐름으로 연결되었기 때문이라고 판단합니다.

## 5. What Was Difficult

### 도메인 안전선 설정

심리상담 도메인은 표현 하나가 오해를 만들 수 있습니다.

AI가 친절하게 말하는 것과 상담사처럼 보이는 것 사이의 경계를 조정하는 것이 어려웠습니다.

### 기능 확장 욕심 관리

RAG, 자원 탐색, 디버그, 보안, STT/TTS 등 확장 아이디어가 많았습니다.

하지만 모든 기능을 현재 버전에 넣으면 프로젝트 범위가 과도해질 수 있었습니다.

따라서 현재 구현 가능한 범위와 후속 과제를 계속 분리해야 했습니다.

### 답변분류 문제

짧지만 의미 있는 답변을 어떻게 처리할지 고민이 있었습니다.

예를 들어 불안, 우울, 오래됐어요 같은 답변은 짧지만 현재 질문에 대해 의미가 있을 수 있습니다.

규칙 기반 분류만으로는 이런 답변을 안정적으로 처리하기 어렵기 때문에 디버그와 LLM fallback 검토가 필요하다고 판단했습니다.

### 문서 정합성 유지

기획서, 기능 정의 보고서, 발표자료, 구현 범위가 서로 어긋나지 않도록 맞추는 작업이 필요했습니다.

특히 구현되지 않은 기능을 이미 구현된 것처럼 보이지 않게 하는 것이 중요했습니다.

## 6. Key Decisions

### 상담 챗봇이 아니라 접수면접 보조 시스템으로 정의

가장 중요한 결정은 서비스 정체성을 좁힌 것입니다.

이 결정 덕분에 프로젝트가 상담 수행 AI가 아니라 접수면접 자동화 및 리포트 정리 PoC로 설명될 수 있었습니다.

### 운영 판단과 임상 판단 분리

AI는 질문을 이어가고, 보완 질문을 하며, 다음 섹션으로 넘어갈 수 있습니다.

하지만 이것은 접수면접 운영을 위한 판단일 뿐, 사용자의 상태를 임상적으로 판단하는 것은 아닙니다.

이 구분은 프로젝트의 핵심 방어선이었습니다.

### 자원 탐색은 추천이 아니라 후보 안내로 제한

상담사나 센터 정보를 보여주더라도 가장 적합한 곳이라고 단정하지 않도록 했습니다.

조건 기반 후보 안내로 제한하면서 최종 선택은 사용자 또는 상담사 확인으로 남겼습니다.

### 디버그는 관찰 레이어로 분리

디버그 모드는 새로운 상담 흐름이 아니라 기존 로직을 관찰하는 레이어로 정의했습니다.

이 덕분에 디버그 기능이 실제 사용자 흐름을 바꾸지 않도록 설계할 수 있었습니다.

## 7. Presentation Feedback

발표 이후 받은 주요 피드백은 다음과 같습니다.

- 3주 미니 프로젝트인데 메인 프로젝트급으로 만들어와서 놀랐다는 평가
- 3주 동안 이 정도까지 만들기 힘들었을 텐데 대단하다는 평가
- 추후 본 프로젝트에 연장해서 가져가도 될 것 같다는 의견
- 기술적인 질문은 별도로 많지 않았음
- 심리상담가 관점에서 TTS와 STT 가능성에 대한 질문이 있었음

이 피드백을 바탕으로, MindBridge는 단기 PoC에서 끝내기보다 본 프로젝트나 포트폴리오 케이스 스터디로 확장할 수 있는 소재라고 판단했습니다.

## 8. Lessons Learned

### 문제 정의가 기능보다 먼저다

처음부터 기능을 많이 넣는 것보다, 어떤 문제를 해결하는지와 어떤 문제는 해결하지 않을지 정하는 것이 중요했습니다.

### AI 프로젝트는 역할 경계가 중요하다

AI가 할 수 있는 일보다 하면 안 되는 일을 정리하는 것이 프로젝트의 신뢰도를 높였습니다.

### 프롬프트는 문장이 아니라 운영 정책이다

프롬프트는 단순히 좋은 답변을 만들기 위한 문장이 아니라, 시스템 역할, 금지 범위, 출력 기준을 통제하는 운영 정책에 가깝다는 것을 배웠습니다.

### 디버그 가능한 구조가 필요하다

LLM 기반 시스템은 문제가 발생했을 때 원인을 찾기 어렵습니다.

따라서 질문 intent, 분류 결과, reason code, chosen action을 확인할 수 있는 구조가 중요하다고 느꼈습니다.

### 후속 과제를 분리하는 것도 기획이다

모든 아이디어를 현재 버전에 넣는 것이 좋은 기획은 아니었습니다.

STT/TTS, LLM fallback, 데이터 갱신 자동화, 운영자 기능 등은 후속 과제로 분리하는 것이 더 안전했습니다.

## 9. If Continued as a Main Project

MindBridge를 본 프로젝트로 확장한다면 다음 방향을 우선 검토할 수 있습니다.

### 1단계. 답변분류 안정화

- 자유서술형 답변 처리 보완
- LLM fallback 도입 범위 검토
- 디버그 로그 분석
- partial 처리 정책 정교화

### 2단계. 리포트 품질 보정

- 상담사 피드백 수집
- 해석형 표현 제거 강화
- 미수집 항목 표시 방식 개선
- 상담소 트랙과 일반인 트랙 리포트 문구 분리

### 3단계. 자원 탐색 데이터 개선

- JSON 및 CSV 정규화 품질 점검
- 미지원 조건 처리 보강
- 데이터 최신성 관리
- 센터 후보와 상담사 후보 출력 방식 개선

### 4단계. 음성 기능 검토

- STT 1차 연동
- TTS 1차 연동
- 음성 인식 결과 확인 단계
- 오인식 수정 UI
- 상담센터 환경 소음 테스트

### 5단계. 운영 및 배포 구조 정리

- 버전 및 빌드 표기
- 운영자용 로그 기능
- 배포 방식 단순화
- 민감정보 처리 기준 정리

## 10. Final Reflection

MindBridge는 기능적으로 완성된 상용 서비스라기보다, AI를 상담 도메인에 적용할 때 어떤 범위와 안전선이 필요한지 검증한 PoC입니다.

이 프로젝트를 통해 AI 서비스 기획에서 중요한 것은 단순히 모델을 붙이는 것이 아니라, 문제 정의, 역할 경계, 실패 가능성, 검증 가능한 구조를 함께 설계하는 것이라는 점을 확인했습니다.

개인적으로는 이 프로젝트를 통해 AI 서비스 기획, 프롬프트 설계, 리스크 관리, 발표 및 문서화 역량을 하나의 사례로 정리할 수 있었습니다.
